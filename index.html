QA Test Rejection Criteria

1. KT Session

  • Reject if major bugs are found during KT session.

2. Dev Test

  • Reject if more than 10 P0 bugs are found within the first 30 minutes of testing.
  • Reject if more than 25 bugs are found when 20% of the project has been tested.
  • Reject if any blockers or functionality breaks are found.
  • Reject if the responsive site is broken on any other devices.

3. HTML Comparison

  • Reject if pages are not aligned with design expectations.
  • Reject if the project is not responsive.
  • Reject if pages are not properly linked.

4. HTML & Dev Retest

  • Reject if all bugs are not labeled correctly (status must be updated to Fixed or relevant status).
  • Reject if more than 5 bugs are reopened.
  • Reject if more than 3 P0 bugs are reopened.
  • Reject if more than 2 regression bugs are found.

5. Before Live Test

  • Reject if any major bugs are found within the first hour of testing.

6. Before Live Retest

  • Reject if more than 1 bug is reopened.

7. After Live

  • Same criteria as Before Live Test.

Bug Review Meeting Criteria

A bug review meeting must be conducted if QA finds:

  • CMS Basic project → More than 40 bugs.
  • CMS Advanced project → More than 50 bugs.
  • E-commerce project → More than 50 bugs.
  • Shopify project → More than 30 bugs.
  • More than 10 repeated bugs.
  • If the project is rejected under any phase.

QA Team Guidelines

1. Attendance & Leave Policy

Office Timing

  • All QA team members must reach the office by 9:30 AM sharp
  • Being on time ensures proper handover of tasks, participation in daily stand-ups, and synchronization with development teams

Planned Leave

  • Submit leave requests at least 2 days in advance to allow proper workload planning
  • Planned leaves must be formally communicated via mail, with prior approval from the QA TL

Emergency Leave

  • In case of emergencies, inform the QA TL first before updating the office group
  • Immediately send a formal leave mail to document the absence
  • Emergency leaves should only be taken for genuine reasons, and frequent usage will be monitored

2. Work Assignments & Guidance

Knowledge Transfer (KT)

  • Before starting a new project, ensure you have taken KT from the Project Coordinator (PC) or the QA TL
  • This avoids misunderstandings, missed requirements, and incomplete coverage

Task Assignment

  • Only the QA TL is authorized to assign tasks
  • Any change in deadlines or additional workload must be documented in the work status report

Work Status Updates

  • Always update the QA Tracker Portal daily
  • Copy and paste the work status exactly as per the portal format to maintain uniformity

3. Task Execution & Deadlines

Timely Completion

  • Tasks must be completed within the assigned timeline
  • Sprint deviations (delays) will be monitored and recorded for performance analysis

If QA TL is Unavailable

  • If no work is assigned by the QA TL due to leave or other reasons:
  • Inform Steve or Sunil immediately
  • Start the project by entering all details in the QA Tracker
  • For GitLab bug tracker setup, seek help from Saneesh, Sunil, or the responsible TL

4. Bug Reporting & Communication

Developer Communication

  • Do not directly debate bugs with developers
  • Escalate clarifications through the responsible TL or Project Coordinator to avoid conflicts and miscommunication

Bug Reporting Guidelines

HTML Comparison

  • Report bugs on Google Doc
  • Make sure to report deviations and complete in 4 hours for 10 pages comparison
  • Use AI tools to make the flow easy
  • Report common bugs in a tab, deviations in a tab

Dev Testing

  • Report bugs on GitLab
  • Report bugs only as individual issues
  • Common bugs should be marked as a single issue
  • All enhancements should be marked as a single issue

Smoke Testing

  • Perform a smoke test within the first hour of testing
  • If the build fails to meet acceptance criteria (rejection criteria), immediately inform the QA TL and reject the build instead of proceeding with full testing

5. Professional Conduct

Workplace Behavior

  • Avoid loose talk or personal conversations on the work floor
  • Maintain professionalism and respect towards colleagues and developers

Quality of Work

  • Ensure all assigned tasks are completed with accuracy and attention to detail
  • A rushed or careless approach will only increase rework and delays

6. Daily Activities

Daily Stand-Ups

  • Attend and complete daily stand-ups before starting your work
  • Share clear updates: yesterday's work, today's plan, and blockers

Hubstaff Tracking

  • Mandatory for all QA members
  • Requirements:
  • Minimum 70% activity rate (to show consistent effort)
  • At least 8 hours of tracked work time daily
  • Low Hubstaff activity without valid reason will be flagged

7. Meetings & Reporting

QA Bug Review Meetings

  • May be conducted by the QA TL when necessary, but only after verification by Steve, Saneesh, or Sunil
  • These meetings help finalize bug statuses, avoid duplication, and track unresolved issues

Project Reporting

  • Every project must follow the standard report format (shared earlier)
  • Reports must include:
  • Project Start Date
  • Expected End Date & Actual End Date
  • Login Credentials
  • Bug Table (with Bug ID, Description, Priority, Status, etc.)
  • Bug Summary (total bugs raised, fixed, pending)
  • Actions to be Taken (recommendations, blockers, pending fixes)

Important Notice

Non-compliance with these guidelines will be taken seriously and may impact performance reviews.
